# Questflow 抓包详细指南

## 一、准备工作

1. 打开 **Chrome** 或 **Edge** 浏览器（推荐 Chrome）
2. 确保已经登录 [https://next.questflow.ai/](https://next.questflow.ai/)
3. 建议先清空之前的网络请求，方便查找

---

## 二、打开开发者工具并监听网络请求

### 步骤 1：打开开发者工具

按 `F12` 或 `Ctrl+Shift+I` (Windows) / `Cmd+Option+I` (Mac) 打开开发者工具。

### 步骤 2：切换到 Network 标签

点击顶部的 **Network** 标签，确保网络监听已开启（红色圆点表示正在录制）。

### 步骤 3：清空之前的请求

点击左上角的 **🚫 清空按钮** (Clear network log)，清除之前的网络请求，方便只看你发送的请求。

### 步骤 4：筛选 Fetch/XHR 请求

点击筛选栏中的 **Fetch/XHR**，只显示 API 请求（过滤掉图片、CSS、JS 等无关请求）。

---

## 三、触发聊天请求并定位 API

### 步骤 5：发送一条测试消息

在 Questflow 的 TARS 聊天窗口中发送一条简单的消息，比如：
```
你好
```

### 步骤 6：查找 API 请求

观察 Network 面板，会新增几个请求。重点关注以下特征：
- **Name**: 可能包含 `chat`, `completion`, `message`, `stream`, `agent`, `tars` 等关键词
- **Method**: 通常是 `POST` 或 `GET`
- **Type**: `fetch` 或 `xhr`

> 💡 **提示**：如果请求太多，可以在 Network 面板的 Filter 框中输入 `chat` 或 `api` 进行过滤。

### 步骤 7：确认正确的请求

点击可疑的请求，在右侧的 **Preview** 或 **Response** 标签中查看返回内容，如果包含 AI 的回复内容，就是正确的请求。

---

## 四、提取关键信息

### 4.1 提取 Cookie（如果请求使用了 Cookie 认证）

点击选中的请求，右侧切换到 **Headers** 标签：

找到 **Request Headers** 部分，找到 `cookie:` 字段，复制整个值。

示例：
```
cookie: _ga=GA1.1.xxx; session=abc123; token=xyz789; ...
```

**完整复制**这个值，填入 `.env` 的 `QUESTFLOW_AUTH_TOKEN`。

### 4.2 提取 API 端点 URL

在 **Headers** 标签中，找到 **General** 部分：

```
Request URL: https://api.questflow.ai/api/v1/chat/completions
```

拆分如下填入 `.env`：
- `QUESTFLOW_BASE_URL` = `https://api.questflow.ai`（基础域名部分）
- `QUESTFLOW_CHAT_ENDPOINT` = `/api/v1/chat/completions`（路径部分）

> ⚠️ 注意：实际的 URL 可能不同，请以你抓到的为准。

### 4.3 提取 Authorization Token（如果使用了 Bearer）

如果在 Request Headers 中看到：
```
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

则：
- `QUESTFLOW_AUTH_TYPE` = `bearer`
- `QUESTFLOW_AUTH_TOKEN` = `eyJhbGciOiJIUzI1NiIs...`（Bearer 后面的部分）

### 4.4 查看请求体（Request Payload）

切换到 **Payload** 标签，查看发送给服务器的 JSON 数据。这是最关键的部分！

示例：
```json
{
  "messages": [
    {"role": "user", "content": "你好"}
  ],
  "model": "tars-default",
  "stream": true,
  "session_id": "abc-123-xyz"
}
```

**记录这个结构**，用于修改 `src/routes/chat.ts` 中的 `buildQuestflowBody` 函数。

### 4.5 查看响应格式（Response）

切换到 **Preview** 或 **Response** 标签，查看服务器返回的数据格式：

- 如果是流式输出，可能会看到多个 `data: {...}` 格式的 SSE 事件
- 如果是非流式，会看到一个完整的 JSON 响应

**记录这个格式**，用于修改 `src/utils/transform.ts` 中的转换函数。

## 五、导出为 cURL（备用方法）

如果你不确定如何提取上述信息，可以导出完整的 cURL 命令：

1. 在 Network 面板中右键点击目标请求
2. 选择 **Copy → Copy as cURL (bash)**
3. 粘贴到文本编辑器中分析

示例 cURL：
```bash
curl 'https://api.questflow.ai/api/v1/chat/completions' \
  -H 'authority: api.questflow.ai' \
  -H 'accept: text/event-stream' \
  -H 'content-type: application/json' \
  -H 'cookie: _ga=GA1.1.xxx; session=abc123;' \
  -H 'origin: https://next.questflow.ai' \
  -H 'referer: https://next.questflow.ai/' \
  --data-raw '{"messages":[{"role":"user","content":"你好"}],"stream":true}'
```

从 cURL 中可以很容易提取：
- URL → `QUESTFLOW_BASE_URL` + `QUESTFLOW_CHAT_ENDPOINT`
- `-H 'cookie: ...'` → `QUESTFLOW_AUTH_TOKEN`
- `--data-raw` → 请求体结构

## 六、修改项目配置

根据抓包结果，修改以下文件：

### 1. 修改 `.env`

```env
QUESTFLOW_BASE_URL=https://api.questflow.ai
QUESTFLOW_AUTH_TYPE=cookie
QUESTFLOW_AUTH_TOKEN=你的完整Cookie字符串
QUESTFLOW_CHAT_ENDPOINT=/api/v1/chat/completions
```

### 2. 修改 `src/routes/chat.ts` 的 `buildQuestflowBody` 函数

根据实际抓到的请求体结构调整字段映射：

```typescript
function buildQuestflowBody(request: OpenAIChatRequest): Record<string, unknown> {
  // 根据你的抓包结果修改这里！
  return {
    messages: convertToQuestflowMessages(request.messages),
    stream: request.stream ?? false,
    model: request.model || "tars-default",
    // 如果 Questflow 需要其他字段，在这里添加
    // session_id: uuidv4(),
    // temperature: request.temperature,
    // max_tokens: request.max_tokens,
  };
}
```

### 3. 修改 `src/utils/transform.ts` 的转换函数（如需要）

如果 Questflow 的响应格式与预设的不同，调整 `convertToOpenAIResponse` 和 `convertToOpenAIStreamChunk` 函数中的字段提取逻辑。

## 七、常见问题

### Q: 没有看到任何 API 请求？
A: 确保你点击了 **Fetch/XHR** 筛选，并且发送消息后才查看 Network 面板。如果用的是 WebSocket，切换到 **WS** 标签查看。

### Q: Cookie 太长，复制不完整？
A: 在 Request Headers 中点击 cookie 字段右侧的 **view source** 或右键 → **Copy value** 来完整复制。

### Q: 请求返回 401/403？
A: Cookie 可能已经过期。重新登录 Questflow，然后再次抓包获取新的 Cookie。

### Q: 响应是加密的或二进制数据？
A: 可能 Questflow 使用了 Protobuf 或其他二进制协议。尝试查看 Response 标签的原始内容，或在 cURL 中添加 `-v` 查看详细输出。

### Q: 使用的是 WebSocket 而不是 HTTP？
A: 如果看到的是 WS 请求，需要额外实现 WebSocket 代理。先检查请求内容确认是否同时有 HTTP API。
